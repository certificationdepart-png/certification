## 3. Деталізація задач

### Модуль A — Бот WayForPay

### WFP-01 · Кнопки керування

- Додати кнопку **«Старт»** для початку роботи з ботом.
- Додати кнопку **«Скасувати»** для скасування поточної дії.

**Критерій приймання:** при запуску бота користувач бачить обидві кнопки; «Старт» ініціює стандартний сценарій, «Скасувати» повертає до початкового стану, скидає заповнені поля.

### WFP-02 · Автоматичне видалення заявок та логів

- Реалізувати автоматичне видалення заявок старших за **3 дні**.
- Реалізувати автоматичне видалення логів старших за **3 дні**.

**Критерій приймання:** записи, створені більше ніж 3 дні тому, автоматично видаляються без ручного втручання. У базі присутні лише актуальні записи за останні 3 дні.

### WFP-03 · Відображення причини неуспішної оплати

- На сторінці «Заявки» додати окремий стовпчик після стовпчика оплати з причиною неуспішної оплати.
- Підтримувані причини (відповідно до WayForPay API):

| # | Причина |
| --- | --- |
| 1 | Недостатньо коштів |
| 2 | Невірні реквізити картки |
| 3 | Ліміт операцій вичерпано |
| 4 | Відхилено банком-емітентом |
| 5 | Сесія користувача минула |
- Статуси оплати мають відображатися коректно відповідно до актуальних даних WayForPay (згідно зі скріншотами, наданими клієнтом раніше).

**Критерій приймання:** менеджер бачить причину відхилення оплати безпосередньо в таблиці заявок без необхідності заходити в кабінет WayForPay.

### WFP-04 · Виправлення оновлення статусів оплати

- Виправити логіку оновлення статусу: якщо клієнт здійснює повторну спробу оплати (після невдалої першої), статус має коректно оновлюватися на актуальний.
- Забезпечити автоматичне оновлення статусу без ручного втручання.

**Критерій приймання:**

| Крок | Дія | Очікуваний результат |
| --- | --- | --- |
| 1 | Клієнт робить першу оплату → відхилена | Статус = «Відхилено», причина відображена |
| 2 | Клієнт робить повторну оплату → успішна | Статус автоматично = «Успішно» |
| 3 | Перевірка без ручного оновлення | Статус коректний без дій менеджера |

### Модуль B — Бот сертифікації

### CRT-01 · Пагінація списку курсів

- Відображення курсів у боті посторінково — **по 7 курсів на сторінку**.
- Кнопки навігації: «Вперед →», «← Назад» для переключення між сторінками.

**Критерій приймання:** список курсів розбитий на сторінки по 7 елементів; кнопки навігації коректно переключають сторінки в обох напрямках.

### CRT-02 · Дані отримувача для Нової Пошти

- Перед запитом населеного пункту додати два додаткових питання:

| # | Поле | Формат | Приклад |
| --- | --- | --- | --- |
| 1 | Прізвище та ім'я отримувача | Вільний текст | Іванов Іван |
| 2 | Телефон отримувача | `380XXXXXXXXX` | 380501234567 |
- Зберігати отримані дані разом із заявкою.

**Критерій приймання:** бот послідовно запитує ПІБ та телефон отримувача перед вибором відділення НП; дані відображаються у відповідній заявці на платформі.

### CRT-03 · Видалення заявок на платформі

- Додати можливість видалення заявок безпосередньо з інтерфейсу платформи (кнопка «Видалити» у рядку заявки), кнопка видалити при вибрані декількох рядків.

**Критерій приймання:** менеджер може видалити заявку з платформи; після видалення заявка зникає зі списку, але залишається в привʼязаної Google Sheet.

### CRT-04 · Збереження заявок на закордонну відправку

- Забезпечити збереження в таблицю інформації по клієнтах, які обирають відправку сертифіката за кордон.

**Критерій приймання:** заявки на закордонну відправку відображаються в таблиці нарівні з іншими, з відміткою типу відправки.

### CRT-05 · Умовне поле «БПР» у заявці

- В налаштуваннях курсу — галочка «Курс містить БПР».
- Якщо увімкнено → студент у заявці обирає «Потрібен» / «Не потрібен».
- Якщо вимкнено → поле не з'являється у заявці.

**Критерій приймання:**

| Налаштування курсу | Поведінка у заявці бота |
| --- | --- |
| БПР ✅ увімкнено | Студент бачить вибір «Потрібен / Не потрібен» |
| БПР ❌ вимкнено | Поле відсутнє, студент його не бачить, також не бачить під час підтвердження заявки |

### CRT-06 · Відкладене повідомлення після відправки сертифіката

- В налаштуваннях курсу додати:

| Елемент | Тип | Опис |
| --- | --- | --- |
| Галочка | Toggle | Увімк/вимк надсилання повідомлення |
| Текст повідомлення | Textarea | Вміст повідомлення студенту |
| Затримка (днів) | Number | 1 = наступний день, 2 = через два дні і т.д. |
- Бот автоматично надсилає повідомлення студенту в Telegram через задану кількість днів після підтвердження менеджером.

**Критерій приймання:** повідомлення надходить студенту через вказану кількість днів; текст і затримка відповідають налаштуванням курсу.

### CRT-07 · Доопрацювання кнопки «Відхилено»

- При натисканні «Відхилено» менеджером:
    1. Відкривається вікно зі списком причин відхилення.
    2. Менеджер обирає причину.
    3. Студенту надходить повідомлення з причиною відхилення.
- В налаштуваннях школи — окремий розділ для управління списком причин (додавання, редагування, видалення).

Базових причин для школи немає, для кожної школи причини та повідомлення для студента для кожної причини налаштовуються окремо.

**Критерій приймання:** при відхиленні менеджер обирає причину зі списку; студент отримує повідомлення з причиною; список причин редагується в налаштуваннях школи.

### Модуль C — Інтеграція Kajabi

### KJB-01 · Виправлення повторного відкриття доступу

- Виправити помилку, при якій повторне відкриття курсу неможливе після видалення попереднього доступу.
- Усунути хибне спрацьовування помилки «email в іншому Kajabi» — забезпечити коректну роботу з email, що існують у кількох акаунтах Kajabi (штатна поведінка Kajabi).

**Критерій приймання:**

| Сценарій | Очікуваний результат |
| --- | --- |
| Відкриття → видалення → повторне відкриття | Доступ відкривається без помилок |
| Email існує в іншому акаунті Kajabi | Доступ відкривається без помилки «email в іншому Kajabi» |
